# /qa-plan — планирование QA-прогона

Команда анализирует `QA_BRIEF.md` и предлагает осмысленный набор QA-агентов для текущей задачи.

## Что делает команда

1. Читает `QA_BRIEF.md` в корне проекта. Если нет — отправляет на `/qa-start`.
2. Читает реестр регрессии `.qa/regression-registry.json` (если есть) — понимает, что уже проверено.
3. Анализирует `git log` последних коммитов — понимает, что менялось.
4. На основе анализа **предлагает набор агентов** для запуска с обоснованием.
5. Пользователь подтверждает или корректирует.
6. После подтверждения — переходит к `/qa-run`.

## Пресеты

Поддерживаются быстрые режимы:

| Пресет | Агенты | Когда использовать |
|--------|--------|--------------------|
| **smoke** | qa-functional + qa-edge-cases | Быстрая проверка после мелких изменений |
| **standard** | qa-functional + qa-consistency + qa-edge-cases + qa-accessibility | Обычный аудит |
| **deep** | Все, кроме qa-security | Перед важным релизом |
| **release** | Все 10 агентов | Перед продакшн-релизом |
| **security** | qa-security + qa-integration | Security-аудит |
| **performance** | qa-performance + qa-accessibility | Проверка метрик перед оптимизацией |
| **regression** | qa-regression + qa-functional | После крупных изменений |
| **bug-hunt** | bug-reproducer + qa-functional + qa-edge-cases | Расследование конкретной жалобы |

Использование:
- `/qa-plan` — умный анализ и рекомендация
- `/qa-plan smoke` — применить пресет smoke
- `/qa-plan custom` — собрать набор руками

## Инструкции для Claude

### Алгоритм

1. **Прочитай `QA_BRIEF.md`.**
2. **Прочитай `.qa/regression-registry.json`** (если есть).
3. **Посмотри git log последних 20 коммитов** — определи области изменений.
4. **Определи контекст прогона:**
   - Первый аудит или повторный?
   - Что менялось с прошлого прогона?
   - Есть ли специфический запрос от пользователя (новая фича, баг-репорт)?

5. **Сгенерируй рекомендацию** в формате:

```
## Рекомендация по QA-прогону

На основе брифа и последних изменений предлагаю:

### Основной набор агентов:
1. ✅ qa-functional — критично: проверить работу основных фич
2. ✅ qa-edge-cases — вы изменили форму регистрации, нужно прогнать edge cases
3. ✅ qa-consistency — добавлены новые страницы, проверить однородность
4. ✅ qa-accessibility — в брифе: WCAG AA обязателен

### Рекомендую пропустить:
- ⏸ qa-performance — метрики проверялись 2 дня назад, новых тяжёлых изменений нет
- ⏸ qa-security — security-аудит недавно прошёл, крупных изменений бэкенда нет
- ⏸ qa-integration — интеграции не трогали

### Не запускаем:
- bug-reproducer — нет активных жалоб от пользователей
- developer — включится автоматически после нахождения багов
- qa-regression — объединится с qa-functional (реестр невелик)

### Предполагаемое время: ~15-25 минут (параллельный прогон)

Подтверждаешь? Можешь изменить набор.
```

6. **Дождись подтверждения** пользователя. Допустимые ответы:
   - "ок" / "запускай" → переход к /qa-run
   - "добавь X" / "убери Y" → обнови рекомендацию
   - "давай smoke" → применить пресет

7. **Сохрани выбранный план** в `.qa/reports/plan-{YYYY-MM-DD-HHmm}.md`:
   ```markdown
   # План QA-прогона {дата}
   
   Выбранные агенты:
   - qa-functional
   - qa-edge-cases
   - ...
   
   Обоснование: ...
   ```

8. **Выведи финальное сообщение:**
```
✅ План согласован.

Запустить сейчас?
- Да → используй /qa-run
- Позже → план сохранён, запустить когда будет готово
```

### Важно

- **Не запускай агентов в этой команде** — только планирование.
- **Адаптируйся под контекст.** Если бриф указывает "стабильность важнее", приоритизируй qa-functional и qa-regression. Если "конверсия важнее" — qa-consistency и qa-accessibility.
- **Объясняй выбор.** Никогда не предлагай агента без обоснования.
- **Предупреждай о дорогих агентах.** qa-performance и qa-security требуют больше времени.
