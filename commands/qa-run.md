# /qa-run — запуск QA-прогона

Команда запускает выбранных агентов параллельно и собирает результаты.

## Варианты использования

```
/qa-run                  # если есть сохранённый план — запуск по нему
/qa-run smoke            # быстрый пресет
/qa-run functional,a11y  # произвольный список (сокращения имён)
/qa-run all              # все 10 агентов
```

## Что делает команда

1. Определяет список агентов для запуска (из аргументов или из сохранённого плана).
2. Проверяет готовность: есть ли `QA_BRIEF.md`? Доступно ли тестовое окружение?
3. **Запускает выбранных агентов параллельно** через Task tool.
4. Собирает результаты каждого агента.
5. Агрегирует баги в единый список.
6. Вызывает `/qa-report` для формирования сводного отчёта.

## Сокращения имён агентов

Чтобы не писать длинные имена:

| Сокращение | Полное имя |
|------------|------------|
| `functional` | qa-functional |
| `regression` | qa-regression |
| `consistency` | qa-consistency |
| `edge` | qa-edge-cases |
| `integration` | qa-integration |
| `perf` / `performance` | qa-performance |
| `a11y` / `accessibility` | qa-accessibility |
| `security` / `sec` | qa-security |
| `repro` | bug-reproducer |
| `dev` | developer |

## Инструкции для Claude

### Алгоритм

1. **Разбери аргументы** команды:
   - Нет аргументов → прочитай `.qa/reports/plan-*.md` (последний)
   - Название пресета → разверни в список агентов (см. `/qa-plan`)
   - Список через запятую → распарси с учётом сокращений

2. **Проверки перед запуском:**
   - `QA_BRIEF.md` существует? Если нет → предложи `/qa-start`.
   - `.qa/` папка существует? Если нет → создай.
   - Staging URL доступен? Сделай `curl -I` — если 404/timeout → предупреди пользователя.
   - Git репо чистый? Если нет — предупреди (не блокер).

3. **Подтверждение запуска:**
```
Готов запустить:
- qa-functional
- qa-edge-cases
- qa-consistency

Проект: Discript Main App
Staging: https://staging.discript.ru
Агенты будут работать параллельно.

Ожидаемое время: 15-25 минут.

Запускаем? (да/нет/изменить)
```

4. **Запуск агентов параллельно.** Используй Task tool с subagent_type для каждого из выбранных агентов. **Все вызовы Task в одном сообщении**, чтобы они работали параллельно.

Пример передачи задачи:
```
Task(
  description: "QA functional audit",
  subagent_type: "qa-functional",
  prompt: "Проведи функциональный аудит проекта согласно QA_BRIEF.md в {path}. 
          Сохрани найденные баги в .qa/bugs/functional/ в формате BUG_REPORT.md.
          После завершения верни сводку: сколько фич проверено, сколько багов найдено."
)
```

5. **Пока агенты работают** — выведи сообщение:
```
🚀 Запущено 3 агента параллельно.
Жди результатов...
```

6. **После завершения всех агентов:**
   - Собери результаты из всех агентов
   - Положи все баги в `.qa/bugs/` с префиксом имени агента
   - Посчитай статистику:
     - Всего багов
     - По серьёзности
     - По категориям
     - По агентам

7. **Выведи короткую сводку:**
```
✅ QA-прогон завершён.

Найдено багов: 18
├─ БЛОКЕРОВ:  1 ⚠️
├─ КРИТИЧНЫХ: 4
├─ ВАЖНЫХ:    9
└─ МЕЛКИХ:    4

Отработали:
├─ qa-functional: 6 багов (1 блокер, 3 критичных, 2 важных)
├─ qa-edge-cases: 7 багов (0 блокеров, 1 критичный, 4 важных, 2 мелких)
└─ qa-consistency: 5 багов (0 блокеров, 0 критичных, 3 важных, 2 мелких)

⚠️ Есть 1 БЛОКЕР — требует немедленного внимания.

Следующий шаг:
- /qa-report — полный отчёт
- /qa-fix — запустить developer на исправление
```

8. **Если агент упал** (не смог закончить работу):
   - Зафиксируй в отчёте
   - Не блокируй остальных
   - Предложи перезапустить упавшего: "Агент qa-security не смог завершить. Перезапустить? (да/нет/пропустить)"

### Важно

- **Параллельность обязательна.** Последовательный запуск слишком долог.
- **Изоляция агентов.** Каждый агент работает независимо. Если один находит что-то, что помогло бы другому — это обрабатывается в следующем цикле, не на лету.
- **Прогресс для пользователя.** Если прогон дольше 5 минут — выводи промежуточные сообщения: "qa-functional закончил, нашёл 6 багов. qa-edge-cases ещё работает."
- **Не запускай `developer`** в этой команде. developer вызывается отдельно через `/qa-fix`.
- **Не запускай `bug-reproducer`** автоматически — он нужен только когда есть конкретный баг от пользователя.
