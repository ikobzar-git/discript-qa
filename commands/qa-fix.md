# /qa-fix — исправление багов

Команда запускает `developer`-агента для исправления багов из отчёта.

## Варианты использования

```
/qa-fix                 # фиксит все блокеры и критичные (дефолт)
/qa-fix blockers        # только блокеры
/qa-fix critical        # блокеры + критичные
/qa-fix all             # вообще все баги
/qa-fix B-001,B-003     # конкретные баги по ID
/qa-fix --dry-run       # показать план фиксов без применения
```

## Что делает команда

1. Читает все баги из `.qa/bugs/`.
2. Фильтрует по указанному критерию.
3. Сортирует по приоритету (блокер → критично → важно → мелкий).
4. Показывает план фиксов.
5. **Запрашивает подтверждение.**
6. Запускает `developer` с планом.
7. После завершения — показывает отчёт о фиксах.

## Инструкции для Claude

### Алгоритм

1. **Определи список багов** для фикса из аргументов:
   - Без аргумента → все с серьёзностью БЛОКЕР и КРИТИЧНО
   - `blockers` → только БЛОКЕРЫ
   - `critical` → БЛОКЕР + КРИТИЧНО
   - `all` → все
   - `B-NNN,B-MMM` → конкретные ID
   - `--dry-run` → план без выполнения

2. **Проверь предусловия:**
   - Git: репозиторий чист? (если нет — предупреди: "Есть незакоммиченные изменения. Фиксы могут смешаться. Закоммить или продолжить?")
   - Есть ли активная ветка для работы? Рекомендуй: "Создадим ветку `qa-fixes/{date}`?" 

3. **Покажи план:**
```
## План исправлений

Будет исправлено: 5 багов

### 🔴 БЛОКЕРЫ (1)
1. **B-001** — Не работает оплата при сумме >10000₽
   - Файл: src/features/checkout/CheckoutForm.tsx:142
   - Изменение: добавить null-check для order.discount
   - Риск: низкий

### 🟠 КРИТИЧНЫЕ (3)
2. **B-002** — Дублирование событий аналитики
   - Файл: src/services/analytics.ts
   - Риск: низкий

3. **B-003** — ...
4. **B-004** — ...

### 🟡 ВАЖНЫЕ (1)
5. **B-012** — Inconsistent modal close behavior
   - Файл: src/components/Modal.tsx
   - Риск: средний (общий компонент)

**Общая оценка изменений:** 5 файлов, ~50 строк кода
**Риски:** средние — один общий компонент (Modal) может затронуть другие места

Применить?  (да / нет / только блокеры / показать детали)
```

4. **При подтверждении** — запусти `developer` агента через Task:

```
Task(
  description: "Fix bugs from QA",
  subagent_type: "developer",
  prompt: """
  Исправь следующие баги, в указанном порядке приоритета:
  
  [полные карточки багов]
  
  Правила:
  - Следуй плану — не добавляй ничего лишнего
  - После каждого фикса давай короткий отчёт
  - Минимум изменений
  - Не ломай остальной код
  
  Верни финальный отчёт о том, что исправлено и что осталось.
  """
)
```

5. **После завершения developer** — выведи сводку:
```
## ✅ Фиксы завершены

Исправлено: 4 из 5
├─ Блокеры: 1/1 ✅
├─ Критичные: 3/3 ✅
└─ Важные: 0/1 ⚠️ (B-012 отложен — требует обсуждения архитектуры Modal)

Изменённые файлы:
- src/features/checkout/CheckoutForm.tsx
- src/services/analytics.ts
- ...

Рекомендуемые следующие шаги:
1. Ревью изменений (git diff)
2. Прогон /qa-run regression — убедиться, что ничего не сломалось
3. Коммит и PR
4. Повторный /qa-run standard для финальной проверки

Хочешь запустить /qa-run regression сейчас?
```

### Важно

- **Никогда не фикси без подтверждения.**
- **Никогда не делай force-push.**
- **Не создавай коммиты без согласия** пользователя — developer вносит изменения, а коммит делает пользователь или по явному указанию.
- **Если фикс требует архитектурных изменений** — останови работу и доложи: "B-012 не фикшу автоматически — требует обсуждения. Хочешь обсудить подход?"
- **Отслеживай git status** перед каждым фиксом и после — чтобы пользователь видел, что именно меняется.

### Безопасные режимы

**`--dry-run`** — показывает полный план с diff-ом, но не применяет изменения.

**`--one-by-one`** — фиксит по одному багу, спрашивая подтверждение после каждого (для паранойи на критичном проекте).
