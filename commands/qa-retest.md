# /qa-retest — повторное тестирование

Команда проводит повторный QA-прогон после фиксов: проверяет, что баги реально исправлены, и ловит возможные регрессии.

## Когда использовать

- После `/qa-fix` — убедиться, что фиксы работают
- Перед релизом — финальная проверка
- При обнаружении жалоб на проде — проверить затронутые области

## Варианты использования

```
/qa-retest              # перепроверить все закрытые баги + регрессия
/qa-retest B-001        # только конкретный баг
/qa-retest full         # полный повторный аудит всеми агентами
```

## Что делает команда

1. Собирает список багов, помеченных как "fixed" в текущей итерации.
2. Для каждого — проверяет повторно по шагам из карточки.
3. Параллельно запускает `qa-regression` для проверки смежного функционала.
4. Формирует отчёт: что реально закрыто, что осталось.

## Инструкции для Claude

### Алгоритм

1. **Собери список** закрытых в этой итерации багов.
2. **Определи режим:**
   - Default: retest закрытых + regression на затронутых
   - Конкретный ID: только он
   - `full`: перезапуск всех агентов, которые работали в прошлом прогоне

3. **Для default-режима** — запусти параллельно:
   - Агент, который нашёл баг изначально → пусть проверит по своему сценарию
   - `qa-regression` → проверка смежного

4. **Формат результата:**
```
## 🔄 Retest — итог

### Проверено фиксов: 4

| ID     | Статус    | Проверил       |
|--------|-----------|----------------|
| B-001  | ✅ закрыт | qa-functional  |
| B-002  | ✅ закрыт | qa-edge-cases  |
| B-003  | ❌ ВЕРНУЛСЯ | qa-functional |
| B-004  | ✅ закрыт | qa-consistency |

### Регрессия: проверено 12 функций

| Функция              | Статус       |
|----------------------|--------------|
| Регистрация          | ✅ работает  |
| Логин                | ✅ работает  |
| Оформление заказа    | ✅ работает  |
| Личный кабинет       | ⚠️ НОВАЯ РЕГРЕССИЯ — B-013 |
| ...                  | ...          |

### Новые баги: 1

**B-013** — Личный кабинет: кнопка "Сохранить" не реагирует после фикса B-003
(связано: фикс B-003 затронул общий компонент, пострадал ЛК)

### Рекомендация

1. Переоткрыть B-003 (вернулся)
2. Исследовать B-013 (новая регрессия как следствие фикса)
3. Повторный /qa-fix на B-003, B-013
4. После — ещё один /qa-retest

Продолжить?
```

5. **Обнови регрессионный реестр** — добавь все проверенные фичи.

### Важно

- **Всегда делай retest одним из способов** — нельзя закрывать итерацию без подтверждения, что фиксы работают.
- **Новые регрессии** часто следствие фиксов. Отмечай связь явно: "B-013 появился после фикса B-003".
- **Если после 2 итераций retest всё ещё находит критичные** — останови цикл и подними вопрос архитектуры.

## Правило 2 итераций

По основному workflow плагина — **максимум 2 полных цикла** аудит-фикс-retest. Если после 2-го retest остались БЛОКЕРЫ или КРИТИЧНЫЕ — это сигнал о более глубокой проблеме, требующей ревью архитектуры или процессов, а не ещё одной итерации.
