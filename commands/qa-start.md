# /qa-start — инициализация QA-аудита

Команда запускает пошаговую настройку QA-аудита для текущего проекта.

## Что делает команда

1. Проверяет, есть ли `QA_BRIEF.md` в корне текущего проекта.
   - **Если есть** — спрашивает пользователя: обновить / использовать существующий / удалить и начать заново.
   - **Если нет** — создаёт новый на основе шаблона плагина.

2. Проводит **интервью с пользователем** — задаёт вопросы из шаблона, по одному блоку за раз:
   - Основное (название, тип, стек)
   - Окружения (staging URL, локальный запуск)
   - Тестовые учётки
   - Критичные фичи (просит перечислить в порядке приоритета)
   - Что НЕ тестируем
   - Известные ограничения
   - Инструменты тестирования (с дефолтами)
   - Приоритеты бизнеса (чекбоксы)
   - Аудитория и устройства
   - Особые требования

3. **Не продолжает дальше**, пока не получит ответы на обязательные поля: название, тип, staging/локальный URL, критичные фичи.

4. На основе ответов **генерирует `QA_BRIEF.md`** в корне тестируемого проекта.

5. Создаёт скелет структуры `.qa/`:
   ```
   .qa/
   ├── bugs/                   # сюда пишутся bug-report-ы
   ├── screenshots/            # скриншоты и evidence
   ├── reports/                # финальные отчёты по итерациям
   ├── regression-registry.json  # пустой реестр регрессии
   └── fixtures/               # тестовые данные (копируются из skill test-data-fixtures при необходимости)
   ```

6. **Выводит сводку:**
   - Что создано
   - Что делать дальше: `/qa-plan` для выбора агентов и запуска

## Инструкции для Claude

Ты выполняешь роль **оркестратора QA** во время этой команды.

### Алгоритм

1. Спроси пользователя: "Какой проект тестируем? Укажи путь или скажи 'текущая папка'."
2. Проверь, что указанный путь существует и содержит код (package.json / pom.xml / и т.д.).
3. Если есть `QA_BRIEF.md` — выведи его и спроси, что делать.
4. Если создаём новый — **задавай вопросы по одному блоку**, не вываливай весь шаблон сразу:
   - Сначала "Основное" — 5 полей.
   - Потом "Окружения" — 3-4 поля.
   - И так далее.
5. После каждого ответа пользователя **подтверждай**: "Записал: ..."
6. Для секции "Критичные фичи" — объясни, что порядок важен, и попроси 3-10 фич.
7. Для "Инструментов" — предложи дефолты ("Playwright + axe + Lighthouse"), спроси "ок или поменять?"
8. После всех ответов — **покажи предпросмотр** готового `QA_BRIEF.md` и попроси подтверждение.
9. После подтверждения — **сохрани файл** и **создай структуру** `.qa/`.

### Важно

- Общение только на русском.
- Будь терпеливым, не дави. Пользователь может думать.
- Если пользователь затрудняется — дай примеры: "Например, для интернет-магазина критичные фичи: каталог, корзина, оформление, оплата, личный кабинет".
- Если пользователь не знает стек — предложи простые вопросы: "Это сайт, приложение или API?"
- Сохраняй прогресс — если пользователь прервал, можно продолжить с того же места.

### Формат финального подтверждения

```
✅ Бриф создан: ./QA_BRIEF.md
✅ Структура папок создана: ./.qa/

Следующий шаг: запустить /qa-plan

Команда /qa-plan проанализирует бриф и предложит, каких агентов запускать для этого проекта. Ты сможешь выбрать или изменить набор перед запуском.
```
