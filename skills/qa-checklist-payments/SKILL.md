# Чек-лист: платежи

> ⚠️ **КРИТИЧНО:** Всё тестирование — ТОЛЬКО в тестовом режиме платёжки (sandbox). Никогда не делай тестовые платежи на production-аккаунтах.

## A. Подготовка

- [ ] Платёжный шлюз в sandbox/test-режиме
- [ ] Используем тестовые карты провайдера (Stripe/ЮKassa/CloudPayments)
- [ ] Тестовый пользователь с чистой историей
- [ ] Известны тестовые карты для разных сценариев (успех, отказ, 3DS)

## B. Отображение корзины/заказа

- [ ] Список товаров корректен
- [ ] Цена каждого товара
- [ ] Количество изменяется
- [ ] Удаление товара работает
- [ ] Применение промокода
- [ ] Расчёт скидки корректен
- [ ] Расчёт доставки корректен
- [ ] Расчёт налога корректен
- [ ] Итоговая сумма = сумма позиций + доставка + налог - скидка
- [ ] Валюта отображается везде одинаково
- [ ] При изменении валюты — конвертация корректна

## C. Ввод платёжных данных

### Карта
- [ ] Поле номера карты: пробелы между группами (4-4-4-4)
- [ ] Автоопределение типа карты (Visa/MC/Mir)
- [ ] Поле срока: ограничение 2 цифры месяц, 2 цифры год
- [ ] Валидация срока (не в прошлом)
- [ ] Поле CVV: 3-4 цифры, `type="password"` или замаскированное
- [ ] Поле имени: латиница / кириллица (по требованию банка)
- [ ] Нельзя ввести буквы в поле номера карты
- [ ] Вставка из буфера с пробелами работает

### Безопасность
- [ ] Все поля защищены HTTPS
- [ ] CVV НЕ сохраняется никогда (даже в логах, даже на секунду)
- [ ] Номер карты маскируется при отображении (•••• 1234)
- [ ] PCI-DSS: форма карты — через платёжку (iframe или redirect), не на твоём домене

### Сохранение карт (tokenization)
- [ ] "Сохранить карту" чекбокс
- [ ] Сохранённая карта отображается как •••• 1234
- [ ] При повторной оплате — выбор из сохранённых
- [ ] Удаление сохранённой карты работает
- [ ] На сервере хранится только токен, не номер карты

## D. Сценарии оплаты

### Успешная оплата
- [ ] Тестовая карта "успех" → платёж проходит
- [ ] Редирект на success-страницу
- [ ] Заказ создан в БД со статусом paid
- [ ] Email-подтверждение пришло
- [ ] Уведомление пришло (push, SMS, если есть)
- [ ] Баланс/остаток обновлён
- [ ] История заказов обновлена

### 3DS (трёхмерная аутентификация)
- [ ] Редирект на страницу банка
- [ ] После подтверждения — возврат с успехом
- [ ] После отказа в 3DS — возврат с ошибкой
- [ ] Таймаут 3DS — корректная обработка
- [ ] Закрытие окна 3DS посередине — корректная обработка

### Отказ банка
- [ ] Тестовая карта "недостаточно средств" → понятное сообщение
- [ ] Тестовая карта "просрочена" → понятное сообщение
- [ ] Тестовая карта "карта заблокирована" → понятное сообщение
- [ ] Сообщения не раскрывают внутренних деталей
- [ ] Заказ остался в статусе pending/failed (не paid!)
- [ ] Деньги не списались

### Отмена пользователем
- [ ] Закрытие вкладки на платёжке → заказ не paid
- [ ] Нажатие "Отменить" на платёжке → возврат с cancel
- [ ] После отмены — можно попробовать снова без пересоздания заказа
- [ ] После отмены — можно создать новый заказ

## E. Webhook-и от платёжки

- [ ] Webhook на успех меняет статус заказа (если ещё не paid)
- [ ] Webhook на отмену меняет статус
- [ ] Webhook на возврат меняет статус
- [ ] Подпись webhook-а валидируется (поддельный не принимается)
- [ ] Идемпотентность: повторный webhook не дублирует действия
- [ ] Out-of-order webhook-и обрабатываются корректно
- [ ] Если webhook не дошёл — есть fallback (опрос статуса или cron)
- [ ] Логирование webhook-ов для отладки

## F. Возвраты (refunds)

- [ ] Пользователь может запросить возврат (если UI)
- [ ] Админ может инициировать возврат
- [ ] Частичный возврат (refund на часть суммы)
- [ ] Полный возврат
- [ ] Возврат отражается в статусе заказа
- [ ] Email-уведомление о возврате
- [ ] Возврат в период chargeback работает
- [ ] Возврат после истечения периода — корректная ошибка

## G. Edge cases

### Суммы
- [ ] Минимальная сумма оплаты (0.01 или 1 ₽)
- [ ] Максимальная сумма (лимит банка)
- [ ] Оплата товаров бесплатно (0 ₽) — пропускаем платёжку или нет?
- [ ] Округление при расчётах (0.333 × 3 ≠ 1.00 без округления)

### Конкурентность
- [ ] Двойной клик "Оплатить" → один платёж, не два
- [ ] Параллельная оплата того же заказа в двух вкладках → один платёж
- [ ] Оплата одного и того же товара из разных корзин

### Временные факторы
- [ ] Платёж прошёл, но страница redirect не открылась → статус корректно обновится по webhook
- [ ] Timeout платёжки → заказ в статусе "ожидает", не в "failed"
- [ ] Retry платежа после ошибки

### Данные
- [ ] Смена email посреди процесса → куда уйдёт чек?
- [ ] Изменение корзины после перехода на оплату → обработка?
- [ ] Исчезновение товара из БД во время оплаты

## H. Специфика для России

- [ ] Тестовая карта Мир работает
- [ ] СБП (Система быстрых платежей) — QR, перевод, возврат
- [ ] Фискальный чек формируется (54-ФЗ)
- [ ] Чек отправляется на email/телефон
- [ ] НДС корректно рассчитан и отображён

## I. Специфика для международных платежей

- [ ] Мультивалютность
- [ ] Конвертация по корректному курсу
- [ ] Комиссия за конвертацию видна
- [ ] Country/Zip-code валидация
- [ ] VAT/Sales tax для разных стран

## J. Безопасность платежей

- [ ] PCI-DSS compliance (собственная форма карты только с сертификатом, иначе через провайдера)
- [ ] Нет логирования полного номера карты
- [ ] Нет логирования CVV нигде и никогда
- [ ] Пользователь не может изменить сумму из DevTools
- [ ] Цена на сервере = цена в заказе (проверка идентичности)
- [ ] Нельзя оплатить чужой заказ (проверка owner на сервере)
- [ ] Количество не может быть отрицательным в запросе
- [ ] Защита от replay attacks (nonce/idempotency key)
