# Чек-лист: авторизация и сессии

## A. Регистрация

### Основной поток
- [ ] Успешная регистрация с валидными данными
- [ ] Редирект после регистрации (куда положено)
- [ ] Автовход после регистрации (если это UX)
- [ ] Письмо-подтверждение приходит (если требуется)
- [ ] Ссылка в письме работает
- [ ] Ссылка в письме истекает через X часов

### Валидация
- [ ] Email должен быть валидным
- [ ] Email уже существует → понятное сообщение
- [ ] Пароль соответствует политике (длина, символы)
- [ ] Подтверждение пароля совпадает
- [ ] Имя/фамилия — валидация длины
- [ ] Согласие с офертой обязательно (если есть)

### Безопасность
- [ ] Пароли не сохраняются в plain text
- [ ] Пароли хешируются bcrypt/argon2, не MD5/SHA1
- [ ] **Enumeration protection:** одинаковое сообщение для "email занят" / "email свободен"
- [ ] Rate limiting на регистрации (не >5 регистраций в минуту с одного IP)
- [ ] CAPTCHA при подозрительной активности
- [ ] Регистрация через disposable email (mailinator) — блокируется, если это нужно бизнесу

## B. Вход (login)

### Основной поток
- [ ] Вход с правильными кредами работает
- [ ] Вход с неправильным паролем → ошибка
- [ ] Вход с неправильным email → такая же ошибка (enumeration protection)
- [ ] Вход с пустыми полями → валидация
- [ ] После входа — редирект на dashboard или returnUrl

### Функциональность
- [ ] "Запомнить меня" работает (срок действия разумный)
- [ ] Без "Запомнить меня" — сессия истекает после закрытия браузера
- [ ] Toggle "показать пароль"
- [ ] Ссылка "Забыли пароль?"
- [ ] Ссылка на регистрацию
- [ ] Вход через соцсети (Google, VK, Telegram — если есть)

### Безопасность
- [ ] Пароль не попадает в URL (GET-параметры, реферер)
- [ ] Пароль передаётся только по HTTPS
- [ ] Rate limiting (блокировка после 5-10 неудач с IP/email)
- [ ] Блокировка аккаунта после N неудач (опционально)
- [ ] Уведомление на email о входе с нового устройства/IP
- [ ] Защита от credential stuffing (CAPTCHA)
- [ ] Session ID меняется после логина (защита от session fixation)

## C. Выход (logout)

- [ ] Logout очищает cookies/localStorage
- [ ] Logout инвалидирует токен на сервере (не просто "забывает" на клиенте)
- [ ] После logout — редирект на публичную страницу
- [ ] После logout back-кнопка не возвращает в авторизованную зону
- [ ] Logout в одной вкладке → другие вкладки тоже разлогиниваются (broadcast)

## D. Восстановление пароля

- [ ] Ссылка "Забыли пароль?" видна
- [ ] Поле email с валидацией
- [ ] После ввода — сообщение "Если такой email есть, на него отправлено письмо" (enumeration protection)
- [ ] Письмо приходит за разумное время (<2 минут)
- [ ] Ссылка в письме работает
- [ ] Ссылка истекает через X минут (обычно 15-60)
- [ ] Ссылка одноразовая (после использования не работает)
- [ ] Форма нового пароля требует подтверждения
- [ ] Новый пароль должен отличаться от старого (опционально)
- [ ] После смены пароля — все активные сессии инвалидируются
- [ ] Rate limiting на reset-request

## E. Смена пароля (в личном кабинете)

- [ ] Требуется ввод текущего пароля
- [ ] Новый пароль + подтверждение
- [ ] Валидация нового пароля (политика)
- [ ] Нельзя поставить тот же пароль, что текущий
- [ ] После смены — уведомление на email
- [ ] После смены — все другие сессии инвалидируются

## F. Сессии и токены

### Хранение
- [ ] JWT в HttpOnly cookie (не localStorage — XSS-риск)
- [ ] Secure flag на cookies (только HTTPS)
- [ ] SameSite=Lax или Strict на auth cookies
- [ ] Refresh token отдельно от access token

### Жизненный цикл
- [ ] Access token короткий (15-60 минут)
- [ ] Refresh token длиннее (дни/недели)
- [ ] Автоматический refresh без выкидывания пользователя
- [ ] При истечении refresh → корректный редирект на логин
- [ ] При истечении access в середине действия → корректная обработка

### Мультисессии
- [ ] Можно быть залогиненным на нескольких устройствах
- [ ] Список активных сессий в профиле (опционально)
- [ ] Возможность завершить все сессии кроме текущей
- [ ] При смене пароля — все сессии завершаются

## G. Роли и права (RBAC)

### Разграничение
- [ ] Страницы/API требуют нужной роли
- [ ] Прямой переход на защищённую страницу без роли → 403 или редирект
- [ ] Кнопки действий скрыты/заблокированы, если нет прав
- [ ] Клиентская проверка дублируется серверной (всегда)

### Тестовые сценарии
- [ ] Войти как обычный юзер → попытаться зайти на /admin → 403
- [ ] Войти как юзер A → попытаться GET /users/B → 403 (IDOR)
- [ ] Войти как юзер A → попытаться DELETE чужого → 403
- [ ] Изменить роль в UI (если манипулируем JS) → сервер не верит клиенту

## H. Двухфакторная аутентификация (2FA)

Если реализована:
- [ ] Включение 2FA требует подтверждения (ввод кода из Authenticator)
- [ ] Backup-коды выдаются при включении
- [ ] При входе — после пароля требуется код
- [ ] Неправильный код → ошибка без раскрытия информации
- [ ] Rate limiting на ввод 2FA-кода
- [ ] Отключение 2FA требует пароль
- [ ] Потеря устройства → восстановление по backup-коду

## I. Социальная авторизация

- [ ] Первичная регистрация через соц-логин создаёт аккаунт
- [ ] Повторный вход — логинит в существующий аккаунт
- [ ] Привязка соц-логина к существующему аккаунту по email
- [ ] Отзыв доступа в соц-сети → корректное поведение при следующем входе
- [ ] Отключение соц-логина в настройках работает
- [ ] Нельзя отвязать последний способ входа (останется способ восстановления)

## J. Специальные проверки

- [ ] Удалённый аккаунт не может войти
- [ ] Заблокированный аккаунт получает понятное сообщение
- [ ] Неподтверждённый email не может войти (если это требование)
- [ ] Замороженный/неактивный аккаунт имеет чёткий статус
- [ ] Логин по username vs email (если оба) — работает одинаково

## K. Edge cases авторизации

- [ ] Вход с истёкшим токеном посередине действия
- [ ] Одновременный вход в двух вкладках
- [ ] Логаут в одной вкладке → поведение в других
- [ ] Смена пароля → все сессии инвалидированы
- [ ] Выход из браузера → возвращение → автологин корректный (или требует пароль)
