# Чек-лист: тестирование форм

## A. Видимость и первое впечатление

- [ ] Все поля видны без скролла (на desktop и mobile)
- [ ] Label каждого поля понятен
- [ ] Обязательные поля помечены явно (* или текстом)
- [ ] Placeholder НЕ заменяет label
- [ ] Автоматический focus на первом поле (если уместно)
- [ ] Группировка связанных полей логична
- [ ] Primary CTA визуально выделен
- [ ] Secondary действия (отмена) менее заметны
- [ ] Адекватные отступы между полями

## B. Ввод данных

### Текстовые поля
- [ ] Принимают корректные данные
- [ ] Работает paste из буфера
- [ ] Работает автодополнение браузера (autocomplete)
- [ ] Поля с email имеют `inputmode="email"` / `type="email"`
- [ ] Поля с телефоном имеют `inputmode="tel"` / `type="tel"`
- [ ] Поля с цифрами имеют `inputmode="numeric"`
- [ ] Password поле имеет `type="password"`
- [ ] Есть toggle "показать пароль" (если UX этого требует)
- [ ] Ограничение длины (maxlength) разумно

### Специальные поля
- [ ] Date picker работает на desktop
- [ ] Date picker работает на mobile (нативный date)
- [ ] Select с поиском (если список длинный)
- [ ] Multi-select понятен
- [ ] Checkbox/radio достаточного размера (≥24x24px)
- [ ] File upload — drag&drop работает
- [ ] File upload — клик по области тоже работает
- [ ] Превью загруженного файла

## C. Валидация

### Логика валидации
- [ ] Валидация происходит в правильный момент (чаще всего onBlur, не onChange)
- [ ] Нет валидации на onChange (раздражает, мешает вводу)
- [ ] Submit блокируется, если есть ошибки
- [ ] Повторная отправка формы блокируется до ответа сервера
- [ ] После исправления ошибки она исчезает
- [ ] Клиентская валидация дублируется серверной

### Сообщения об ошибках
- [ ] Ошибки показываются рядом с полем (не только общим списком)
- [ ] Сообщения конкретные ("Email должен содержать @", а не "Неверно")
- [ ] Сообщения на языке пользователя
- [ ] Сообщения с `role="alert"` / `aria-live` для скрин-ридеров
- [ ] Поле с ошибкой имеет `aria-invalid="true"`
- [ ] Ошибка визуально выделена (бордер + текст, не только цвет)

### Типовые проверки
- [ ] Email: есть ли `@` и домен
- [ ] Phone: длина и формат для выбранной страны
- [ ] Password: соответствует политике (длина, символы)
- [ ] Password confirm: совпадает с основным
- [ ] Required: нельзя отправить пустым
- [ ] Maxlength: нельзя ввести больше
- [ ] Pattern: формат соблюдён

## D. Отправка формы

- [ ] Кнопка submit блокируется на время запроса
- [ ] Есть индикатор загрузки (спиннер, disabled state)
- [ ] Двойной клик не отправляет дважды
- [ ] При успехе — явное сообщение + редирект (если нужно)
- [ ] При ошибке сервера — понятное сообщение, форма не очищается
- [ ] Таймаут обрабатывается (не висит бесконечно)
- [ ] При потере сети — корректное сообщение

## E. Навигация и состояние

- [ ] Back-кнопка не ломает состояние
- [ ] Refresh — данные сохраняются или есть предупреждение
- [ ] Уход со страницы с несохранёнными данными → предупреждение
- [ ] Отмена — форма очищается/возвращается в исходное состояние
- [ ] Многошаговая форма — можно вернуться к предыдущему шагу
- [ ] Прогресс в многошаговой форме виден

## F. Edge cases

### Граничные значения
- [ ] Ввод ровно на границе maxlength
- [ ] Ввод на 1 символ больше maxlength (обрезается / показывает ошибку?)
- [ ] Поле длиной 1 символ (достаточно?)
- [ ] Очень длинное содержимое (10000 символов)

### Спецсимволы
- [ ] Unicode (эмодзи, китайский, арабский)
- [ ] HTML-теги: `<script>`, `<img>`
- [ ] SQL-паттерны: `' OR 1=1 --`
- [ ] JSON: `{"a":1}` в текстовом поле
- [ ] Пробелы в начале/конце (trim?)
- [ ] Только пробелы (трактуется как пусто?)

### Данные
- [ ] Email с плюсом: `a+tag@example.com`
- [ ] Email с поддоменом: `user@mail.example.com`
- [ ] Email на кириллице: `пример@почта.рф`
- [ ] Телефон с разными форматами: `+7 (999) 123-45-67`, `89991234567`, `+79991234567`
- [ ] Имя с дефисом: `Анна-Мария`
- [ ] Имя с апострофом: `O'Brien`

## G. Доступность

- [ ] Все поля имеют видимый label (не только placeholder)
- [ ] Label связан с полем через `for`/`id`
- [ ] Tab-навигация обходит все поля в логичном порядке
- [ ] Enter в текстовом поле → submit (не перенос, если это не textarea)
- [ ] Enter в textarea → перенос строки (не submit)
- [ ] Focus-индикатор виден на всех полях
- [ ] Ошибки анонсируются скрин-ридером
- [ ] Обязательные поля помечены `required` или `aria-required`

## H. Mobile

- [ ] На мобилке нет горизонтального скролла формы
- [ ] Поля достаточно большие для тапа (≥44x44px)
- [ ] При фокусе на поле — экран не прыгает
- [ ] Клавиатура соответствует типу поля (цифровая для чисел)
- [ ] Submit-кнопка не закрывается клавиатурой
- [ ] Автозаполнение работает

## I. Performance

- [ ] Форма рендерится быстро (<500ms)
- [ ] Валидация не лагает при каждом вводе
- [ ] Submit не блокирует весь UI
